# Identity System

The identity system solves one of the hardest problems in multi-agent collaboration: **how does an LLM know which agent it is, especially when multiple accounts of the same model are participating?**

---

## The problem

Consider a setup with two Codex accounts:
- `orchestrator` — primary orchestrator
- `worker` — secondary implementation agent (also Codex)

Without explicit identity management:
1. You open a new Codex session for `worker`
2. It has no idea it's `worker`, not `orchestrator`
3. It might read messages meant for `orchestrator`
4. It might respond as if it were `orchestrator`
5. You have to verbally tell it "you are worker" — which you might forget

The `llm-collab` identity system eliminates all of these failure modes.

---

## How it works

### 1. Per-agent directory

Every LLM agent has a dedicated directory:

```
agents/
├── orchestrator/
│   ├── identity.md      ← who this agent is
│   ├── memory.md        ← persistent memory (owned by this agent)
│   └── inbox.json       ← unread message pointers
├── worker/
│   ├── identity.md      ← explicitly states identity + inbox scope
│   ├── memory.md
│   └── inbox.json
```

Inboxes are scoped per agent. `worker` can only read messages addressed to `worker`.

### 2. Identity file read first at bootstrap

When an agent runs `session_bootstrap.py`, the **first output** is the contents of `agents/{id}/identity.md`:

```
══════════════════════════════════════════════════════════════
IDENTITY
══════════════════════════════════════════════════════════════
# Identity: Worker

> **IMPORTANT**: You are Worker (worker). Read only messages addressed to 'worker'.

## Role
**Agent ID**: `worker`
...
══════════════════════════════════════════════════════════════
```

The LLM reads this before anything else. No verbal instruction required.

### 3. Human relay handoff prompts

For agents that require a human to start a new LLM session (`activation.type: "human_relay"`), the system generates the handoff prompt automatically whenever anyone sends them a message:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠  Worker requires human relay.
   Share this prompt with the operator to activate them:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

You are Worker (worker). Read only messages addressed to 'worker'.

Bootstrap your session by running:
  python /path/to/_collab/bin/session_bootstrap.py --agent worker

Then read your inbox and execute your latest task.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

The operator copies this and pastes it into a new LLM session. The LLM immediately knows:
- Its exact identity
- What it is NOT
- The exact command to bootstrap with

## Configuring agents

### Standard LLM agent (cli_session)

```json
{
  "id": "orchestrator",
  "display_name": "Orchestrator",
  "role": "primary_orchestrator",
  "activation": {
    "type": "cli_session",
    "watcher_enabled": true
  }
}
```

No handoff prompt generated. Watcher runs in background. Session is persistent.

### Same-model second account (human_relay)

```json
{
  "id": "worker",
  "display_name": "Worker",
  "role": "implementation",
  "activation": {
    "type": "human_relay",
    "watcher_enabled": false,
    "base_model": "codex",
    "identity_note": "You are Worker (worker). Read only messages addressed to 'worker'."
  }
}
```

`identity_note` appears in the handoff prompt and in `identity.md`. Keep it explicit and unambiguous.

### Human operator

```json
{
  "id": "operator",
  "display_name": "Operator",
  "role": "human_dispatcher",
  "activation": {
    "type": "human",
    "watcher_enabled": false
  }
}
```

No identity file generated. No watcher. Human reads directly.

## Identity file anatomy

Generated by `scripts/init.py` at `agents/{id}/identity.md`:

```markdown
# Identity: Worker

> **IMPORTANT**: You are Worker (worker). Read only messages addressed to 'worker'.

## Role

**Agent ID**: `worker`
**Role**: implementation
**Workspace**: my-collab

## Workspace

- Root: `/path/to/_collab`
- Your inbox: `/path/to/_collab/agents/worker/inbox.json`
- Your memory: `/path/to/_collab/agents/worker/memory.md`

## Session Bootstrap

At the start of every session, run:
```
python /path/to/_collab/bin/session_bootstrap.py --agent worker
```

## Key Commands

```bash
# Check inbox
python /path/to/_collab/bin/inbox.py --me worker

# Send message
python /path/to/_collab/bin/deliver.py --chat last --from worker --to <agent> --title "..."

# Create task
python /path/to/_collab/bin/new_task.py --title "..." --created-by worker --project <project_id>

# Task board
python /path/to/_collab/bin/task_board.py
```

## Active Projects

my-app, my-api

## Other Agents

- `orchestrator`
- `researcher`
- `operator`

## Instructions

- Always bootstrap your session before starting work.
- Always check your inbox before starting new work.
- When you complete a task, update its status.
- When you send to a human_relay agent, the system prints a handoff prompt automatically.
```

To regenerate after adding new agents or projects:

```bash
python scripts/init.py
```

Init is idempotent — it only generates missing files.

## Memory files

`agents/{id}/memory.md` is owned by that specific agent. It persists notes across sessions.

The agent is responsible for maintaining it. No other agent should write to it.

```markdown
# Memory: Worker

## Notes

- Project my-app uses pnpm, not npm
- Orchestrator prefers PRs over direct merges
- Auth middleware is in src/middleware/auth.ts

## Completed work

- 2026-04-07: Implemented checkout API (TASK-ABC123)
```

## Generating LLM-specific memory snippets

Use `init_agent_memory.py` to make any LLM aware of the workspace from the start of a session:

```bash
# Claude Code — writes to ~/.claude memory system
python bin/init_agent_memory.py --agent claude --target claude-code --write

# Codex — prints to console, paste into Codex memory
python bin/init_agent_memory.py --agent codex --target codex

# Any LLM — universal markdown block
python bin/init_agent_memory.py --agent orchestrator --target generic

# Inject into a project CLAUDE.md
python bin/init_agent_memory.py --agent claude --target claude-md \
  --project-path ~/Projects/my-app --write
```

The generated snippet includes the workspace path, agent ID, bootstrap command, key commands, and project list. When the LLM loads this at session start, it already knows the workspace exists and how to use it — even without running bootstrap first.

## Frequently asked questions

**Q: What if I forget to run session_bootstrap?**

The LLM won't know its identity or see its inbox. Always run bootstrap. Adding the bootstrap command to your LLM's global memory file (via `init_agent_memory.py`) makes it a habit.

**Q: Can two agents have the same base model?**

Yes. That's exactly what human_relay is for. `codex` and `codex2` can both be Codex instances — they just have different `id` values, different `identity.md` files, and different inboxes. The `identity_note` in the human_relay config should explicitly scope the receiver to its own ID.

**Q: What if an agent updates its own identity.md?**

That's fine — identity files are gitignored and agent-owned. If you add new projects or agents to the workspace, regenerate with `python scripts/init.py`.

**Q: Can I have more than two accounts of the same model?**

Yes. Add `worker2`, `worker3`, etc. Each gets its own identity, inbox, and memory. Keep each `identity_note` explicit, for example: `You are Worker2 (worker2). Read only messages addressed to 'worker2'.`
